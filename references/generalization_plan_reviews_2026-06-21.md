# 泛化方案三轮计划审核

日期：2026-06-21

## 计划初稿

1. 在现有分支增加分镜角色 YAML、图片风格、TTS Provider 和动画配置。
2. 保留原 Mask 页面及兼容逻辑，避免影响已有项目。
3. 修复字幕时间轴和渲染问题后发布体验分支。

## 第一次审核：版本基线

问题：

- 初稿默认当前 Mask 实现是最新版本，实际分支仍停留在 `manual_mask_exact_v2`。
- “保留兼容逻辑”会继续暴露旧覆盖率、红色诊断和最终预览，直接违背最新交互。

修订：

- 先合并 `manual_mask_boundary_white_v4`，再做泛化。
- 旧自动扩框、母版拆层、覆盖率门禁只会继续制造双轨逻辑，应从生产路径和 UI 删除。

## 第二次审核：配置边界

问题：

- 仅提供一个“自定义规则”文本框，用户看不到角色清单、必选项和 JSON Schema。
- 全局修改 pipeline profile 会影响所有项目，测试风险过大。

修订：

- 配置改为项目级 `planning/pipeline_profile.yaml`。
- 同一弹窗暴露 Prompt 规则、结构 YAML、JSON Schema 和完整请求。
- 分镜编辑器直接允许切换角色、添加和删除视觉结构块。

## 第三次审核：可验证性与渲染闭环

问题：

- 动画如果只出现在前端下拉框，没有进入 manifest、timeline 和 Remotion，属于无效功能。
- 只跑单元测试不能证明 8001 的真实项目可以渲染。
- 颜色校验阈值没有基于当前编码输出校准，会把成功渲染误判为失败。

最终计划：

1. 动画字段贯通 UI、manifest、scene builder、Schema、timeline 和 Remotion。
2. 每次渲染前强制重建 Mask v4 资产。
3. 对当前项目完成真实 Remotion 渲染并读取四页解码 MAE。
4. 保留视频颜色元数据硬校验，将画面 MAE 阈值按实测调整。
5. 完成静态检查、回归检查和浏览器端到端检查后再提交推送。
